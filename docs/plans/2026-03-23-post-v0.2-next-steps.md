# DecisionAtlas Post-v0.2 Next Steps

Date: 2026-03-23
Status: Current planning baseline

## Where The Project Is Now

The project has moved past MVP construction and past the main `v0.2` public-demo hardening slices.

What is done:

- demo vs imported provenance is explicit
- real repository ingest includes selected high-signal markdown documents
- live repository analysis has a stable imported-workspace path
- the guided demo lane is now the primary product narrative
- homepage, dashboard, review, why-search, timeline, and drift support a coherent walkthrough
- no active OpenSpec changes remain

This means the project is now in:

```text
post-v0.2 hardening
        ↓
release / packaging / next-slice planning
```

## Completed Plan Summary

The completed plan up to this point is:

1. Build the core MVP:
   - import
   - extraction
   - review
   - why-search
   - timeline
   - drift
2. Harden `v0.2` for public credibility:
   - provenance labeling
   - stronger real repo ingest
   - hardened live repository analysis
   - guided demo lane stabilization
3. Archive the completed OpenSpec changes and sync them into main specs

This phase should be treated as complete unless a regression appears.

## Recommended Next Work Order

### 1. Release-quality cleanup

Before starting the next capability slice, finish a short cleanup pass:

- keep `README.md`, quick-start, and demo-script aligned
- run the release checklist end to end
- decide which local-only files should remain intentionally untracked

This keeps the current baseline trustworthy.

### 2. Real functionality slice

The next major change should prioritize the real product path instead of more demo-only polish.

Recommended change theme:

```text
strengthen-real-repository-outcomes
```

Suggested scope:

- run end-to-end live-provider analysis on several public repositories
- improve how imported workspaces turn repository evidence into useful accepted decisions
- reduce the number of runs that end in thin outputs without clear next actions
- make the real why-answer and drift path more operational

Concrete goals:

- more imported workspaces should yield reviewable candidate decisions
- accepted decisions from real imports should produce believable why-answers
- drift should become easier to evaluate after real imports, either manually or with clearer operator flow

Why this should come next:

- the guided demo is already stable
- the biggest remaining product risk is whether the real lane can complete meaningful repo-analysis tasks
- this work produces the evidence needed before deciding how much to invest in hosted delivery

### 3. Hosted public demo slice

After the real-functionality slice, the next distribution change should be:

```text
host-public-demo
```

Suggested scope:

- deployment environment layout
- health checks
- hosted smoke validation
- operator notes for demo resets
- safer handling of live provider credentials

This should happen after the real path is stronger, not before.

### 4. Start v0.3 only after the above

The current `v0.3` direction is still valid:

- GitHub App auth
- private repository support
- login and roles
- workspace scoping
- richer benchmark corpus

But `v0.3` should not start until:

- the demo is stable
- live-provider validation is credible
- hosted/demo delivery is clear

## What Not To Do Next

Avoid these immediately:

- adding broad new connectors
- redesigning the whole drift system without usage evidence
- mixing hosted-demo work with auth/private-repo platform work
- continuing ad hoc edits without a new change proposal

## Suggested Decision

If no new regression appears, the next concrete move should be:

1. run a release-style cleanup pass
2. open a new OpenSpec change for real repository outcome improvement

That keeps the project coherent:

```text
stable demo
  → stronger real analysis
  → publishable demo
  → platform features
```
