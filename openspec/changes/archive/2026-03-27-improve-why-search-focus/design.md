## Context

The current why-search path is already functional for imported workspaces:

```text
question
  -> rewrite_query()
  -> hybrid_search() over accepted decisions
  -> top hits
  -> stitched answer + citations
```

Recent real-repository validation showed that the system can now answer imported why-questions with citations, but it still has a clear quality problem: answers can blend adjacent accepted decisions into one response even when the user is asking about a single decision. This makes the result feel like a retrieval summary rather than a focused explanation.

The current trust boundary is good and should be preserved:

- imported why-search is still grounded in accepted decisions
- citations still come from source refs
- the product does not need a second free-form answer model call to improve this stage

## Goals / Non-Goals

**Goals:**

- Make why-search prefer a single primary accepted decision when the question clearly points to one decision.
- Tighten when secondary hits are allowed into the answer.
- Improve query normalization so wording differences do not push the system toward unrelated neighboring decisions.
- Keep citation-first trust boundaries intact.
- Add targeted regression coverage for "串题" cases observed in imported workspaces.

**Non-Goals:**

- Replace why-search with unconstrained generative answering.
- Redesign artifact chunk retrieval or make chunk retrieval the main why path.
- Change imported why-search to work without accepted-decision grounding.
- Rework drift, indexing, or extraction logic beyond what is needed to support focused why answers.

## Decisions

### Decision: Select a primary decision before composing the answer

The retrieval path will move from "merge a small set of high-scoring hits" to "pick one primary accepted decision first, then decide whether any secondary decision is eligible to support it."

Why:

- the observed product issue is not missing answers, but multi-decision blending
- a primary-decision-first model maps better to the user mental model of asking one why-question about one decision

Alternatives considered:

- keep current top-hit plus threshold merge
  - rejected because it is the direct source of over-merged answers
- force every answer to use only one hit with no exceptions
  - rejected because some questions legitimately need one supporting related decision

### Decision: Make secondary-hit inclusion semantic and asymmetric

Secondary hits will only be included when they reinforce the same rationale thread as the primary hit, rather than merely scoring above a numeric threshold.

Why:

- "related" and "supporting" are not the same thing
- many imported repositories contain tightly clustered release or migration decisions that overlap lexically but should remain separate in answers

Alternatives considered:

- lower the merge threshold and hope for fewer blends
  - rejected because threshold tuning alone is too brittle
- use all retrieved hits but label them more clearly
  - rejected because that preserves the core focus problem

### Decision: Strengthen query rewrite before changing answer tone

The change will improve normalization and alias handling before spending effort on more natural answer prose.

Why:

- retrieval quality is still the main driver of answer focus
- better wording alone does not solve the wrong-hit or extra-hit problem

Alternatives considered:

- rewrite answer composition first
  - rejected because it would polish an unstable hit set

### Decision: Keep answer composition deterministic and citation-first

The focused answer will still be assembled from accepted decision fields and source refs rather than generated by a second unconstrained LLM step.

Why:

- the current trust boundary is a product strength
- this change is about focus and selection, not inventing a new answer architecture

Alternatives considered:

- add a second LLM summarization pass
  - rejected because it expands scope and risks weaker grounding

## Risks / Trade-offs

- **[Risk] Primary-hit selection becomes too narrow and drops legitimate supporting context** → Mitigation: allow one supporting decision when it passes a stricter semantic support rule and benchmark both single-hit and supported-hit cases.
- **[Risk] Query normalization introduces accidental alias collisions** → Mitigation: start with a small technical synonym set and benchmark against existing imported why questions.
- **[Risk] Answer focus improves but recall regresses for broad questions** → Mitigation: keep separate handling for obviously broad or comparative questions in tests and benchmark fixtures.
- **[Risk] Deterministic composition still sounds mechanical** → Mitigation: accept that trade-off in this change and defer prose improvements until retrieval focus is stable.

## Migration Plan

1. Add focused retrieval selection logic without changing the accepted-decision trust boundary.
2. Expand query rewrite with a limited synonym and normalization layer.
3. Update answer composition to emit a primary answer plus optional supporting context.
4. Add targeted imported-repo regression cases for multi-decision blending.
5. Compare the new path on existing imported repository examples before widening further.

## Open Questions

- Should broad "why did we change release process" style questions deliberately allow more than one supporting decision?
- Should support eligibility use only retrieval features, or should it also inspect overlap in citations and source artifacts?
- How aggressively should the system hide secondary hits versus exposing them as "related context" outside the main answer body?
